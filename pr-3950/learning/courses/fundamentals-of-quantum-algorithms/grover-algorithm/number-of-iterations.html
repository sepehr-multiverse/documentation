Choosing the number of iterations

We have established that the state vector of the register Q\mathsf{Q}Q in Grover's algorithm remains in the two-dimensional subspace spanned by ∣A0⟩\vert A_0\rangle∣A0​⟩ and ∣A1⟩\vert A_1\rangle∣A1​⟩ once the initialization step has been performed.

The goal is to find an element x∈A1,x\in A_1,x∈A1​, and this goal will be accomplished if we can obtain the state ∣A1⟩\vert A_1\rangle∣A1​⟩ — for if we measure this state, we're guaranteed to get a measurement outcome x∈A1.x\in A_1.x∈A1​. Given that the state of Q\mathsf{Q}Q after ttt iterations in step 2 is

Gt∣u⟩=cos⁡((2t+1)θ)∣A0⟩+sin⁡((2t+1)θ)∣A1⟩,G^t \vert u \rangle = \cos\bigl((2t + 1)\theta\bigr) \vert A_0\rangle + \sin\bigl((2t + 1)\theta\bigr) \vert A_1\rangle,Gt∣u⟩=cos((2t+1)θ)∣A0​⟩+sin((2t+1)θ)∣A1​⟩,

⟨A1∣Gt∣u⟩=sin⁡((2t+1)θ)\langle A_1 \vert G^t \vert u \rangle = \sin((2t + 1)\theta)⟨A1​∣Gt∣u⟩=sin((2t+1)θ)

is as close to 111 as possible in absolute value, to maximize the probability to obtain x∈A1x\in A_1x∈A1​ from the measurement. For any angle θ∈(0,2π),\theta \in (0,2\pi),θ∈(0,2π), the value sin⁡((2t+1)θ)\sin((2t + 1)\theta)sin((2t+1)θ) oscillates as ttt increases, though it is not necessarily periodic — there's no guarantee that we'll ever get the same value twice.

Naturally, in addition to making the probability of obtaining an element x∈A1x\in A_1x∈A1​ from the measurement large, we would also like to choose ttt to be as small as possible, because ttt applications of the operation GGG requires ttt queries to the function f.f.f. Because we're aiming to make sin⁡((2t+1)θ)\sin( (2t + 1) \theta)sin((2t+1)θ) close to 111 in absolute value, a natural way to do this is to choose ttt so that

(2t+1)θ≈π2.(2t + 1) \theta \approx \frac{\pi}{2}.(2t+1)θ≈2π​.

Solving for ttt yields

Of course, ttt must be an integer, so we won't necessarily be able to hit this value exactly — but what we can do is to take the closest integer to this value, which is

t=⌊π4θ⌋.t = \Bigl\lfloor \frac{\pi}{4\theta} \Bigr\rfloor.t=⌊4θπ​⌋.

This is the recommended number of iterations for Grover's algorithm. As we proceed with the analysis, we'll see that the closeness of this integer to the target value naturally affects the performance of the algorithm.

(As an aside, if the target value π/(4θ)−1/2\pi/(4\theta) - 1/2π/(4θ)−1/2 happens to be exactly half-way between two integers, this expression of ttt is what we get by rounding up. We could alternatively round down, which makes sense to do because it means one fewer query — but this is secondary and unimportant for the sake of the lesson.)

Recalling that the value of the angle θ\thetaθ is given by the formula

we see that the recommended number of iterations ttt depends on the number of strings in A1.A_1.A1​. This presents a challenge if we don't know how many solutions we have, as we'll discuss later.

Unique search

First, let's focus on the situation in which there's a single string xxx such that f(x)=1.f(x)=1.f(x)=1. Another way to say this is that we're considering an instance of the Unique search problem. In this case we have

Unknown number of solutions

If the number of solutions s=∣A1∣s = \vert A_1 \verts=∣A1​∣ is unknown, then a different approach is required, for in this situation we have no knowledge of sss to inform our choice of t.t.t. There are, in fact, multiple approaches.

One simple approach is to choose

t∈{1,…,⌊πN/4⌋}t \in \Bigl\{ 1,\ldots,\bigl\lfloor\pi\sqrt{N}/4\bigr\rfloor \Bigr\}t∈{1,…,⌊πN​/4⌋}

uniformly at random. Selecting ttt in this way always finds a solution (assuming one exists) with probability greater than 40%, though this is not obvious and requires an analysis that will not be included here. It does makes sense, however, particularly when we think about the geometric picture: rotating the state of Q\mathsf{Q}Q a random number of times like this is not unlike choosing a random unit vector in the space spanned by ∣A0⟩\vert A_0\rangle∣A0​⟩ and ∣A1⟩,\vert A_1\rangle,∣A1​⟩, for which it is likely that the coefficient of ∣A1⟩\vert A_1\rangle∣A1​⟩ is reasonably large. By repeating this procedure and checking the outcome in the same way as described before, the probability to find a solution can be made very close to 1.1.1.

There is a refined method that finds a solution when one exists using O(N/s)O(\sqrt{N/s})O(N/s​) queries, even when the number of solutions sss is not known, and requires O(N)O(\sqrt{N})O(N​) queries to determine that there are no solutions when s=0.s=0.s=0.

The basic idea is to choose ttt uniformly at random from the set {1,…,T}\{1,\ldots,T\}{1,…,T} iteratively, for increasing values of T.T.T. In particular, we can start with T=1T = 1T=1 and increase it exponentially, always terminating the process as soon as a solution is found and capping TTT so as not to waste queries when there isn't a solution. The process takes advantage of the fact that fewer queries are required when more solutions exist. Some care is required, however, to balance the rate of growth of TTT with the probability of success for each iteration. (Taking T←⌈54T⌉T \leftarrow \lceil \frac{5}{4}T\rceilT←⌈45​T⌉ works, for instance, as an analysis reveals. Doubling T,T,T, however, does not — this turns out to be too fast of an increase.)

The trivial cases
